# Visual Design Rules

These rules apply to the UI flow design document around the product sketch, not necessarily to the product application drawn inside the sketch.

## UI Flow Document Style

- Use a clear design-board layout: left tree navigation, compact explanation, large page sketch.
- Keep the navigation narrow but readable. It should expose chapters and page nodes without competing with the sketch.
- Keep the explanation panel readable for product logic, state rules, edge cases, and user intent, but smaller than the sketch area.
- Keep the sketch area large enough to inspect a product page sketch.
- Use a professional, quiet product-design presentation style.
- Use consistent spacing, chapter numbering, and page titles so the flow is scannable.
- Show one active page at a time. Do not stack every page vertically in the default view.
- Avoid marketing hero sections unless the requested product page itself is a marketing page.
- Avoid decorative orbs, bokeh, and generic gradients.

## Product Sketch Style

- Generate the visual style inside each right-side sketch from the user's product requirements.
- Do not force one fixed app style across all products.
- A SaaS/admin tool should be dense, structured, and operational.
- A consumer/mobile product may be more expressive if the user asks for that.
- Keep text inside the sketch readable and non-overlapping.

## Responsive Behavior

- Desktop: persistent left tree navigation plus active-page content. Inside the active page, use a smaller description column and larger sketch column.
- Mobile/tablet: move the tree navigation into a drawer. Inside the active page, stack description above sketch.
- The mobile drawer should close after selecting a page.
- Preserve stable sketch dimensions with responsive constraints.
- Avoid horizontal overflow.
